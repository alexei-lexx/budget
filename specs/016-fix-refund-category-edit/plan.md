## Summary

**Bug**: When editing a refund transaction with an assigned category, the category field appears empty and gets lost on save.

**Root Cause**: The type watcher in [TransactionForm.vue:140-150](../../frontend/src/components/transactions/TransactionForm.vue#L140-L150) clears the categoryId when it detects type mismatch. For refunds, it compares `selectedCategory.type (EXPENSE)` !== `formData.value.type (REFUND)`, which evaluates to true, causing the category to be cleared.

**Fix**: Update the type watcher to use the mapped type (EXPENSE for REFUND) when comparing category compatibility, matching the logic already used in `filteredCategories` computed property.

## Phase 0: Research & Analysis

**Decision**: SKIPPED - No research needed

**Rationale**:
- Bug root cause clearly identified in [TransactionForm.vue:140-150](../../frontend/src/components/transactions/TransactionForm.vue#L140-L150)
- Fix approach is straightforward: replicate type mapping logic from `filteredCategories` (lines 47-51)
- No new technologies, patterns, or unknowns
- No architecture decisions required

**Technical Analysis**:

The bug occurs because:
1. Refund transactions use EXPENSE categories (business rule)
2. `filteredCategories` correctly maps REFUND → EXPENSE for filtering (lines 47-51)
3. **TWO watchers** have the same bug (lines 126-137 and 140-150):
   - Account change watcher (line 132)
   - Type change watcher (line 145)
4. Both compare `selectedCategory.type !== formData.value.type` without mapping
5. Result: `selectedCategory.type (EXPENSE) !== formData.value.type (REFUND)` → category cleared

**Solution**:
Apply the same type mapping in BOTH watchers:
```typescript
// Before: selectedCategory.type !== formData.value.type
// After:  selectedCategory.type !== (formData.value.type === "REFUND" ? "EXPENSE" : formData.value.type)
```

**Affected Lines**:
- Line 132: Account change watcher
- Line 145: Type change watcher
