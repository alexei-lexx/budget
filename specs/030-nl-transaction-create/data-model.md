# Data Model: Natural Language Transaction Creation

**Phase 1 output** | **Date**: 2026-03-03 | **Plan**: [plan.md](plan.md)

---

## Overview

This feature introduces no new persistent entities. All data is stored in the existing `Transaction`, `Account`, and `Category` DynamoDB tables. The only new data structures are:

1. A GraphQL input type (`CreateTransactionFromTextInput`) accepted by the new mutation
2. An internal agent output schema (structured JSON inferred by the AI agent) — not persisted

---

## Entities Involved

### Transaction (existing)

The created entity. No field additions or removals.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | `string` (UUID) | Yes | Generated by repository |
| `userId` | `string` | Yes | From authenticated context — never from input |
| `accountId` | `string` | Yes | Resolved by agent from user accounts |
| `categoryId` | `string` | No | Resolved by agent from user categories; omitted when no match |
| `type` | `TransactionType` | Yes | `EXPENSE`, `INCOME`, or `REFUND` — inferred by agent |
| `amount` | `number` | Yes | Extracted by agent from text |
| `currency` | `string` | Yes | Derived from resolved account (not inferred independently) |
| `date` | `DateString` (YYYY-MM-DD) | Yes | Parsed by agent from text; defaults to today |
| `description` | `string` | No | Derived by agent from meaningful text tokens |
| `isArchived` | `boolean` | Yes | Always `false` on creation |

**Type constraint**: Only `NonTransferTransactionType` values (`EXPENSE`, `INCOME`, `REFUND`) can be created through this mutation. Transfer types (`TRANSFER_IN`, `TRANSFER_OUT`) are not inferrable from free text.

### Account (existing, read-only)

Used by the agent for currency matching and account resolution. Not mutated.

| Field | Used By Agent | Notes |
|-------|--------------|-------|
| `id` | Yes | Output as `accountId` in agent JSON |
| `name` | Yes | Used for display in reasoning |
| `currency` | Yes | Currency filter (FR-005 step 2) |
| `isArchived` | Yes | Agent sees all accounts; service defers filtering to `TransactionService` |

### Category (existing, read-only)

Provided to the agent as the full list to select from. Not mutated.

| Field | Used By Agent | Notes |
|-------|--------------|-------|
| `id` | Yes | Output as `categoryId` in agent JSON |
| `name` | Yes | Agent matches text semantically to category names |
| `type` | Yes | Agent must match category type to inferred transaction type |
| `isArchived` | Yes | Agent receives all categories; active-only filtering is existing repository behavior |

## Internal Data Flow

### GraphQL Input

```
CreateTransactionFromTextInput {
  text: String!           ← free-text user input (e.g. "spent 45 euro at rewe yesterday")
}
```

### Agent Tool Set

| Tool | Type | Purpose |
|------|------|---------|
| `getAccounts` | read | Retrieve all user accounts (id, name, currency) for account resolution |
| `getCategories` | read | Retrieve all user categories (id, name, type) for category matching |
| `getTransactions` | read | Retrieve recent transaction history for most-used account inference |
| `createTransaction` | write | **Terminal action** — agent calls this when all fields are resolved; returns full transaction object (same pattern as data tools) |

### `createTransaction` Tool Input (Zod-validated before reaching TransactionService)

```typescript
{
  type: "EXPENSE" | "INCOME" | "REFUND",
  amount: number,        // positive
  accountId: string,
  categoryId?: string,   // omitted when no category in user's list is a reasonable match
  date: string,          // YYYY-MM-DD
  description?: string,  // omitted when nothing meaningful extracted
}
```

### Agent Flow

```
[text input]
  → Agent: call getAccounts, getCategories, getTransactions to gather context
  → Agent: reason over all fields (type, amount, date, description, category, account)
  → Agent: call createTransaction({...resolved fields})
    → Tool: transactionService.createTransaction(input, userId) → Transaction persisted
    → Tool: returns JSON.stringify(createdTransaction) to agent
  → Agent: final answer = { "transaction": { "id": "..." } }
  → Service: parses transaction.id from response.answer, calls transactionService.getTransactionById(id, userId)
  → Service: returns Transaction
```

### Error Path (unresolvable required fields)

```
[text input: e.g. "bought something" with no amount]
  → Agent: gathers context, cannot determine amount
  → Agent: does NOT call createTransaction
  → Agent: final answer = plain-text explanation
  → Service: JSON.parse(response.answer) fails or yields no transaction.id → throw BusinessError(response.answer)
  → Resolver: handleResolverError → BAD_USER_INPUT GraphQL error
```

### Service Input Delegated to TransactionService (via tool)

```typescript
transactionService.createTransaction(
  {
    accountId: string,
    amount: number,
    categoryId?: string,
    date: DateString,
    description?: string,
    type: NonTransferTransactionType,
  },
  userId,
)
```

All existing validation in `TransactionService.createTransaction` (account ownership, category validation, amount validation) is applied.

## State Transitions

No new state machines. The created transaction is immediately active (`isArchived: false`).

```
[text input]
  → AI Agent: call getAccounts, getCategories, getTransactions
  → AI Agent: reason over all fields
  → AI Agent: call createTransaction tool({...resolved fields})
    → Tool: transactionService.createTransaction(input, userId) → Transaction persisted
    → Tool: returns JSON.stringify(createdTransaction) to agent
  → Agent final answer: { "transaction": { "id": "..." } }
  → Service: parses transaction.id, fetches via transactionService.getTransactionById(id, userId)
  → Service: returns Transaction
```

## Validation Rules

**Layer 1 — Service (before agent call):**

| Order | Check | Owner | Error |
|-------|-------|-------|-------|
| 1 | `userId` must be non-empty | `CreateTransactionFromTextService` | `INVALID_PARAMETERS` |
| 2 | `text` must be non-empty after trim | `CreateTransactionFromTextService` | `INVALID_PARAMETERS` |

**Layer 2 — `createTransaction` tool (Zod + TransactionService, inside agent loop):**

| Order | Check | Owner | Error |
|-------|-------|-------|-------|
| 3 | Tool input validated by Zod schema (type, amount, accountId, date are required) | `createTransaction` tool | Tool returns error string back to agent |
| 4 | Account ownership, category validation, amount range | `TransactionService.createTransaction` | Propagated as `BusinessError` |

**Error path when agent skips `createTransaction`:**

| Trigger | Detection | Error surfaced |
|---------|-----------|----------------|
| Agent cannot resolve amount, account, or sees multiple transactions | `response.answer` is plain text (not `{ transaction: { id } }` JSON) | `BusinessError` with `response.answer` as message |

## No Schema Migrations Required

No DynamoDB table changes. No new GSIs, attributes, or table definitions needed. The existing `Transaction` table fully supports the created records.
