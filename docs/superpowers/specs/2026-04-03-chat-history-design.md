# Chat History Design

**Date:** 2026-04-03
**Issue:** alexei-lexx/budget#291

## Overview

Add conversation history to the Insight service and Telegram integration so users can ask follow-up questions. History is bounded by N recent messages and auto-expires via DynamoDB TTL.

## Goals

- In-app Insight and Telegram bot both support multi-turn conversations
- Each in-app Insight instance and each Telegram chat get their own session
- History: last 20 messages (configurable via env var)
- Message expiry: 24 hours (configurable via env var), implemented as DynamoDB TTL per message
- `InsightService` stays stateless — extensible for future Telegram capabilities (e.g. transaction creation from text)

## Out of Scope

- UI to explicitly reset a session
- Chat history for `CreateTransactionFromTextService` (designed to accommodate it later)
- Routing multiple Telegram capabilities (Insight vs. transaction creation) — but architecture must not prevent it

---

## Service Architecture

Session management is a separate concern from `InsightService`. A new `InsightChatService` handles the in-app chat lifecycle. `ProcessTelegramMessageService` handles the Telegram chat lifecycle. Both use a shared `ChatMessageRepository`. `InsightService` is kept stateless.

```
GraphQL Mutation Resolver  ──▶  InsightChatService  ──▶  InsightService (stateless)
                                        │                        ▲
                               ChatMessageRepository             │
                                        │                        │
ProcessTelegramMessageService  ─────────┘────────────────────────┘
```

### Why not a single shared chat wrapper?

`ProcessTelegramMessageService` is the natural Telegram orchestrator — it already owns Telegram-specific routing and will eventually route to multiple capabilities (Insight, transaction creation, etc.) sharing the same session history. Merging it with an in-app service would conflate two distinct entry points.

## Data Model

### New DynamoDB Table: `ChatMessagesTable`

| Attribute | Type | Notes |
|---|---|---|
| `userId` | String (PK) | Internal app user ID |
| `sessionMessageCombinedId` | String (SK) | `${sessionId}#${messageId}` — ULID provides sort order within session |
| `role` | String | `ChatMessageRole` enum: `"USER"` or `"ASSISTANT"` |
| `content` | String | Message text |
| `createdAt` | String | ISO 8601 timestamp |
| `expiresAt` | Number | Unix timestamp = `createdAt + 24h` — also serves as the DynamoDB TTL attribute |

**Session identity:**
- In-app: UUID generated by backend on first message
- Telegram: `"${botId}#${chatId}"` (deterministic, no creation step needed)

**Loading history:** Query by `userId` with `begins_with(sessionMessageCombinedId, "${sessionId}#")`, take last 20. No TTL filtering in code — DynamoDB handles cleanup in the background.

**Soft-deletion exception:** Chat messages expire via DynamoDB TTL instead of `isArchived` flag. Messages are immutable and short-lived; TTL-based cleanup is the intended lifecycle mechanism.

## New Types

### `Ulid` branded type

```typescript
// backend/src/types/ulid.ts
export type Ulid = string & { readonly __brand: unique symbol };

export function generateUlid(): Ulid {
  return ulid() as Ulid;
}
```

Follows the existing `DateString` pattern. ULID is chosen specifically for lexicographic sortability — making this explicit in the type prevents accidentally passing a plain UUID where sort order matters.

## Repository Layer

### Port

```typescript
// backend/src/services/ports/chat-message-repository.ts

export interface CreateChatMessageInput {
  userId: string;
  sessionId: string;
  messageId: Ulid;
  role: ChatMessageRole;
  content: string;
  createdAt: string;
  expiresAt: number;
}

// Soft-deletion exception: messages expire via DynamoDB TTL instead of isArchived.
// They are immutable and short-lived; TTL-based cleanup is the intended lifecycle.
export interface ChatMessageRepository {
  findManyRecentBySessionId({ userId, sessionId, limit }: {
    userId: string;
    sessionId: string;
    limit: number;
  }): Promise<ChatMessage[]>;

  create(input: CreateChatMessageInput): Promise<ChatMessage>;
}
```

### Implementation

**File:** `backend/src/repositories/dyn-chat-message-repository.ts`

- Table name from `CHAT_MESSAGES_TABLE_NAME` env var
- `findManyRecentBySessionId`: `QueryCommand` with `KeyConditionExpression = "userId = :userId AND begins_with(sessionMessageCombinedId, :prefix)"` where prefix = `"${sessionId}#"`. Sort ascending. Take last `limit` items. Hydrate each record via Zod schema.
- `create`: `PutCommand` with `sessionMessageCombinedId = "${sessionId}#${messageId}"`.

**Schema:** `backend/src/repositories/schemas/chat-message.ts` — Zod schema typed to the raw DynamoDB record shape (not `ChatMessage` directly). The record has `sessionMessageCombinedId` instead of separate `sessionId`/`messageId` fields; all other fields including `expiresAt` map directly. The repository extracts `sessionId` and `messageId` from `sessionMessageCombinedId` when transforming the validated record into a `ChatMessage` entity.

**Tests:** `backend/src/repositories/dyn-chat-message-repository.test.ts` — real DynamoDB Local connection. `describe` blocks mirror method order.

## Service Layer

### `InsightService` — gains `history` param

```typescript
// Existing service, minimal change
async call(userId: string, input: {
  question: string;
  history?: AgentMessage[];  // new optional param
}): Promise<InsightOutput>
```

History is prepended to the messages array sent to the agent. No session awareness — service remains stateless.

### `InsightChatService` — new, in-app chat orchestrator

**File:** `backend/src/services/insight-chat-service.ts`

Single-purpose service (`call` method).

```typescript
async call(userId: string, input: {
  question: string;
  sessionId?: string;
}): Promise<Result<{ answer: string; sessionId: string; agentTrace: AgentTraceMessage[] }>>
```

Flow:
1. If no `sessionId` → generate UUID
2. Load last 20 messages via `ChatMessageRepository.findManyRecentBySessionId`
3. Map to `AgentMessage[]` (role + content)
4. Call `InsightService.call(userId, { question, history })`
5. On success: `create` user message + assistant message (each with `expiresAt = now + 24h`)
6. Return `{ answer, sessionId, agentTrace }`

Dependencies injected: `InsightService`, `ChatMessageRepository`.

**Tests:** `backend/src/services/insight-chat-service.test.ts` — mocked repository.

### `ProcessTelegramMessageService` — gains history support

Existing service extended:
- Add `ChatMessageRepository` dependency
- Derive `sessionId = "${botId}#${chatId}"`
- Load last 20 messages before calling `InsightService`
- Save user + assistant messages after response (with `expiresAt = now + 24h`)

Future Telegram capabilities (e.g. transaction creation from text) plug into the same service using the same session history — no changes to `InsightService` or `ChatMessageRepository` needed.

## GraphQL Schema

`Query.insight` is removed. Replaced with:

```graphql
type Mutation {
  askInsight(input: AskInsightInput!): AskInsightOutput!
}

input AskInsightInput {
  question: String!
  sessionId: ID
}

union AskInsightOutput = AskInsightSuccess | AskInsightFailure

type AskInsightSuccess {
  answer: String!
  sessionId: ID!
  agentTrace: [AgentTraceMessage!]!
}

type AskInsightFailure {
  message: String!
  agentTrace: [AgentTraceMessage!]!
}
```

`sessionId` is optional in input (absent = new session created by backend). Required in success output — frontend stores it for follow-up messages.

## Frontend

The in-app Insight component switches from `Query.insight` to `Mutation.askInsight`:

- On mount: read `sessionId` from `localStorage`
- On send: call `askInsightMutation({ question, sessionId })`
- On success: write returned `sessionId` to `localStorage`, display answer
- On reopen: `sessionId` restored from `localStorage` → conversation continues
- After 24h: messages TTL out server-side; stored `sessionId` still valid — next message starts with empty history (transparent to user)

## Future Considerations

- **Transaction creation via Telegram:** `ProcessTelegramMessageService` gains routing logic to call `CreateTransactionFromTextService` (instead of always calling `InsightService`). Both services accept `history?: AgentMessage[]` — follow-up context (e.g. "same as previous but 5.6") works because prior exchanges are in the shared history.
- **`CreateTransactionFromTextService`:** Will gain `history?: AgentMessage[]` param following the same pattern as `InsightService`.
